# Demo reset для локальной базы данных

## Назначение demo reset

`make demo-reset` полностью очищает прикладные данные локальной базы ResourceFlow и заново заполняет её воспроизводимым демонстрационным набором.

Команда нужна для:

- быстрого возврата базы к известному состоянию без тестового мусора;
- подготовки среды для показа интерфейсов, фильтров, правил бронирования и разных статусов;
- повторяемой локальной проверки сценариев администратора, менеджера и сотрудника.

## Требования перед запуском

Перед запуском должны быть выполнены следующие условия:

- `APP_ENV=development` или `APP_ENV=local`;
- локальный PostgreSQL настроен в соответствии с backend `.env`;
- backend-миграции уже применены;
- задана переменная `DEMO_SEED_PASSWORD`.

## Запуск в PowerShell

Из корня репозитория:

```powershell
$env:DEMO_RESET_CONFIRM='YES'
$env:DEMO_SEED_PASSWORD='your-password'
make demo-reset
```

Для Unix-shell команда остаётся такой же:

```bash
DEMO_RESET_CONFIRM=YES DEMO_SEED_PASSWORD='your-password' make demo-reset
```

## Защитные ограничения

- Команда отказывается запускаться вне `development` / `local`.
- Для запуска требуется явное подтверждение: `DEMO_RESET_CONFIRM=YES`.
- Пароль демонстрационных пользователей читается только из `DEMO_SEED_PASSWORD`.
- Очистка и повторное заполнение выполняются в одной транзакции.

## Что удаляется и что сохраняется

Во время `make demo-reset` удаляются все прикладные данные целевой локальной БД:

- подразделения;
- пользователи и их роли;
- категории ресурсов;
- типы ресурсов;
- правила бронирования;
- ресурсы;
- интервалы `resource_unavailability`;
- legacy-интервалы `resource_availability`;
- бронирования.

При этом сохраняются:

- история миграций;
- служебная структура схемы;
- справочник ролей;
- сами миграционные файлы.

## Представленные варианты правил бронирования

В demo-наборе есть:

- правило без ограничения по времени суток — например для ноутбука и проектора;
- правило с рабочим окном `07:00–22:00` — например для легкового автомобиля;
- правило с обязательным подтверждением — для конференц-зала, транспорта и части групповых ресурсов;
- правило без подтверждения — для переговорной, проектора и ноутбука;
- разные минимальные длительности — `5`, `30` и `60` минут;
- разные горизонты бронирования — `10`, `14`, `21`, `30` и `60` дней;
- единый лимит активных бронирований — `20` для всех 8 демонстрационных правил;
- отключённое правило для типа `Рабочее место`, чтобы в admin-интерфейсе было видно предупреждение об отсутствии действующего правила.

## Интервалы недоступности ресурсов

Созданы понятные интервалы `resource_unavailability`:

- временное закрытие переговорной «Нева» для настройки оборудования;
- техническое обслуживание ГАЗель Next;
- ремонт проектора Epson EB-FH52;
- завершённые прошлые интервалы для корректной демонстрации истории и завершённых состояний.

Набор специально включает оба сценария:

- ресурсы без дополнительных ограничений по недоступности;
- ресурсы с одним или несколькими техническими окнами.

## Бронирования и серия множественного бронирования

Создаётся демонстрационный набор бронирований со статусами:

- `pending`;
- `confirmed`;
- `completed`;
- `cancelled`;
- `rejected`.

Также есть серия множественного бронирования:

- один и тот же ресурс `Lenovo ThinkPad T14`;
- одинаковое время;
- одинаковая цель;
- три последовательные даты;
- все записи валидны и не пересекаются.

Будущие `pending` и `confirmed` подобраны так, чтобы:

- оставаться в будущем после запуска backend worker;
- попадать в рабочее время правила;
- не пересекаться с `resource_unavailability`;
- не пересекаться друг с другом;
- не нарушать лимиты активных бронирований в демонстрационном наборе.

Прошедшие бронирования создаются сразу в конечных статусах.

## Сценарии демонстрации через интерфейс

### Администратор

- Войти под `anna.smirnova@resourceflow.example`.
- Открыть каталог ресурсов и проверить фильтры по типам, категориям, активности и доступности бронирования.
- Найти `Рабочее место A-17` и увидеть предупреждение об отсутствии действующего правила.
- Открыть `Booking Rules` и показать разные комбинации правил: с подтверждением, без подтверждения, с рабочим окном и без него.
- Открыть `Users` и убедиться, что неактивный пользователь присутствует в списке.

### Менеджер

- Войти под `mikhail.volkov@resourceflow.example`.
- Открыть `/bookings` и увидеть `pending`, `confirmed` и терминальные статусы.
- Показать подтверждение и отклонение заявок.
- Найти бронирования по цели, пользователю и ресурсу.

### Сотрудник

- Войти под `elena.kuznetsova@resourceflow.example`.
- Открыть каталог ресурсов и показать поиск по названию, описанию, локации и типу.
- Перейти в карточку ресурса и показать правило бронирования, недоступные интервалы и серию множественного выбора дат.
- Открыть `Мои бронирования` и показать фильтрацию по статусам.

## Важное предупреждение

`make demo-reset` удаляет **все прикладные данные** целевой локальной БД и заново создаёт демонстрационный набор.

Используйте эту команду только для локальной среды, где такая очистка допустима.
