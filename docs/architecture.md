# Архитектура проекта ResourceFlow

## Назначение документа

Документ описывает общую архитектуру системы, состав основных компонентов и принципы взаимодействия между ними.

---

## Архитектурный подход

Система ResourceFlow строится по клиент-серверной архитектуре.

Основные части системы:
- frontend;
- backend;
- база данных PostgreSQL;
- Redis как вспомогательный инфраструктурный компонент.

Такой подход позволяет разделить:
- пользовательский интерфейс;
- бизнес-логику;
- хранение данных.

---

## Общая схема системы

### Frontend
Frontend отвечает за:
- отображение страниц;
- взаимодействие с пользователем;
- отправку HTTP-запросов к backend;
- отображение результатов операций.

Технологии:
- HTML;
- CSS;
- JavaScript.

---

### Backend
Backend отвечает за:
- аутентификацию пользователей;
- проверку прав доступа;
- обработку бизнес-логики;
- управление ресурсами;
- управление доступностью;
- управление бронированиями;
- управление интервьюированием;
- журналирование действий;
- формирование данных для отчетности.

Технология:
- Go.

---

### PostgreSQL
PostgreSQL используется как основное хранилище данных.

В базе данных хранятся:
- пользователи;
- роли;
- подразделения;
- категории и типы ресурсов;
- ресурсы;
- интервалы доступности;
- бронирования;
- кандидаты;
- этапы интервью;
- обратная связь;
- журнал действий.

---

### Redis
Redis используется как вспомогательный компонент.

Возможные направления применения:
- кэширование;
- хранение временных данных;
- ускорение отдельных операций.

На этапе MVP использование Redis может быть минимальным.

---

## Архитектура backend

Backend реализуется как модульный монолит.

Это означает:
- приложение запускается как единый сервис;
- внутри приложения логика разделена на отдельные модули и слои;
- отсутствует избыточная сложность микросервисной архитектуры.

---

## Основные backend-модули

В системе выделяются следующие основные модули:
- аутентификация;
- пользователи;
- подразделения;
- ресурсы;
- доступность ресурсов;
- правила бронирования;
- бронирования;
- кандидаты;
- интервью;
- журналирование;
- отчетность.

---

## Слои backend

### Handler layer
Отвечает за:
- прием HTTP-запросов;
- валидацию входных данных;
- формирование HTTP-ответов.

### Service layer
Отвечает за:
- основную бизнес-логику;
- проверки правил;
- обработку сценариев системы.

### Repository layer
Отвечает за:
- доступ к PostgreSQL;
- выполнение SQL-запросов;
- работу с Redis при необходимости.

### Model / DTO layer
Отвечает за:
- доменные структуры;
- структуры запросов и ответов API.

---

## Взаимодействие компонентов

Основной сценарий взаимодействия:
1. Пользователь выполняет действие в интерфейсе.
2. Frontend отправляет HTTP-запрос к backend.
3. Backend выполняет валидацию и бизнес-логику.
4. Backend обращается к PostgreSQL и при необходимости к Redis.
5. Backend формирует JSON-ответ.
6. Frontend отображает результат пользователю.

---

## Почему выбран такой подход

Выбранная архитектура позволяет:
- упростить реализацию MVP;
- сохранить понятную структуру проекта;
- избежать избыточной сложности;
- обеспечить удобство сопровождения;
- легче объяснить систему в дипломной работе.

---

## Вывод

Для дипломного проекта архитектура модульного монолита с REST API, PostgreSQL и простым frontend является наиболее сбалансированным решением между практичностью и архитектурной чистотой.
